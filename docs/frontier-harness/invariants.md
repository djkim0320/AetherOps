## 신뢰 경계

- system/repository/user/project policy가 external content보다 우선한다.
- 웹, PDF, MCP/provider 응답, tool output, child-agent output은 명령이 아니라 data다.
- raw prompt, provider response, hidden reasoning, token, cookie, secret은 trace, FTS, embedding, SSE, report에 저장하지 않는다.
- artifact와 evidence는 stable handle로 참조하고 promotion 전에 origin, hash, verifier를 확인한다.
- filtering과 authorization은 retrieval 후 renderer나 memory에서만 수행하지 않고 storage query와 실행 직전에 모두 적용한다.

## M1 provider-neutral context 정책

- `ContextProviderCapabilityReceipt`는 SDK 타입이 아닌 provider-neutral profile이다. strict structured-output 지원 여부와 native context/compaction의 `derived_cache_only`, `canonicalStateAuthority=false` 정책을 명시하고 profile version 및 canonical SHA-256으로 결합한다.
- compiler와 persistence readback은 capability profile hash를 다시 계산한다. provider 또는 model ID가 같아도 profile version/content가 바뀌면 다른 `ContextPack` hash가 생성되며, 위조된 receipt는 fail-closed 처리한다.
- optional recent-conversation window는 최대 16개/32,000 UTF-16 code unit의 비정규화 derived cache일 뿐이다. 사용 시 각 entry body hash를 검증하고 secret을 redaction한 뒤 prompt에만 사용할 수 있으며 persistence receipt에는 ID/hash/selection만 기록한다.
- production 기본 경로는 recent-conversation body를 만들지 않는다. 검증된 hash-bearing `conversation_memo` artifact handle만 `snapshot.conversation_artifacts` selector를 통해 전달한다. optional body window 지원은 provider-neutral adapter 평가를 위한 의도적인 M1 deviation이며 resume source-of-truth나 transcript 저장 경로가 아니다.

## 목표 rollout (M7 deferred)

- 현재 production research handler에는 아래 mode switch가 구현되어 있지 않다. M1 canonical wrapper가 무조건 연결되어 있으므로 legacy prompt parity와 project opt-in은 잔여 호환성 위험이다.
- 목표 `legacy`: `a0727f2` routing/memory 결과를 유지하며 trace/eval instrumentation만 허용한다.
- `shadow`: vnext context/tool/memory selection을 계산하고 receipt로 비교하지만 side effect는 legacy 경로만 실행한다.
- `vnext`: held-out/adversarial promotion gate를 통과하고 명시적으로 opt-in한 project에서만 활성화한다.
- rollback은 schema/data 삭제나 downgrade가 아니라 active release pointer 전환이다.
- 기존 RPC/SSE는 additive schema로 확장하고 renderer server state는 TanStack Query만 소유한다.
- 기존 `.aetherops`에는 temp-root migration check/apply/verify/reapply/rollback이 통과하기 전 쓰지 않는다.

## 단계 완료 규칙

각 milestone은 before-state evidence, failing reproduction 또는 missing-capability test, implementation, targeted verification, compatibility impact, rollback path를 갖는다. 실패한 단계가 있으면 다음 milestone의 빈 interface, placeholder table, 미사용 UI를 추가하지 않는다.
